# Review: 02 - Implementation Plan

**Reviewer:** Cursor Agent (600f2a13)
**Plan doc:** `docs/plans/codex-integration/02-implementation.md`
**Review date:** 2026-05-10

## Verdict

REQUEST CHANGES

## Summary

The implementation plan identifies the right starting files for a basic Codex adapter: `hook.rs` and `main.rs`. It is unsafe as written because it branches too late, after error paths that Codex treats as fail-open, and because it serializes from a `Permission` while current hook behavior is represented by Claude-shaped `HookOutput`.

## Detailed Findings

### Finding 1: Existing error propagation would fail open in Codex

**Severity:** Critical
**Location:** Change 6: Update `run()` function
**Claim:** The change can reuse existing evaluation and branch at the output boundary.
**Reality:** The current hook returns `Result<()>` and uses `?` on stdin parsing, engine construction, permission evaluation, and stdout serialization. In Codex, those failures produce non-zero exits or no valid deny envelope, which Codex treats as fail-open. The relevant paths are `crates/permit0-cli/src/cmd/hook.rs:491-498`, `crates/permit0-cli/src/cmd/hook.rs:544-583`, `crates/permit0-cli/src/cmd/hook.rs:671-672`, and `crates/permit0-cli/src/main.rs:210-277`.
**Recommendation:** Split Codex execution into a wrapper that converts recoverable errors into a valid Codex block. Do not let `?` escape after the client is known to be Codex.

### Finding 2: `codex_output(Permission, reason)` cannot represent the current hook decision model

**Severity:** Major
**Location:** Change 4, Change 6, Remote mode, Shadow mode, Unknown mode
**Claim:** Codex output can be generated by mapping `Permission` to empty stdout or deny JSON.
**Reality:** The current hook's final behavior is not a `Permission`. Unknown policy rewrites `HookOutput` in `crates/permit0-cli/src/cmd/hook.rs:355-374`, remote responses become `HookOutput` in `crates/permit0-cli/src/cmd/hook.rs:315-337`, remote errors become `HookOutput` in `crates/permit0-cli/src/cmd/hook.rs:417-428`, and shadow mode returns `HookOutput::allow()` in `crates/permit0-cli/src/cmd/hook.rs:653-667`.
**Recommendation:** Add a client-neutral internal decision type such as `HookDecision { decision: Allow | Deny | Ask | Defer, reason, is_unknown }`. Apply unknown policy and shadow mode to that type, then serialize separately for Claude and Codex.

### Finding 3: Remote HITL handling is already wrong for the daemon's actual response

**Severity:** Major
**Location:** Remote mode
**Claim:** The HTTP call is unchanged and the same remote response can be translated through Codex output.
**Reality:** The daemon emits `"human"` for HITL because `Permission::HumanInTheLoop` displays as `HUMAN` and `serve.rs` lowercases it in `crates/permit0-cli/src/cmd/serve.rs:577-589`. The hook remote mapper expects `"humanintheloop"` only in `crates/permit0-cli/src/cmd/hook.rs:315-337`. This means a real daemon HITL response currently falls into the unknown-permission branch.
**Recommendation:** Fix the remote mapper to accept `"human"` and update the unit test that currently pins `"humanintheloop"` as the "real" response shape.

### Finding 4: Remote session continuity requires daemon/session work if it remains in v1

**Severity:** Major
**Location:** Files NOT Changed
**Claim:** No daemon or session changes are required.
**Reality:** The daemon can parse `metadata.session_id`, but `check_handler` only creates an empty session context in `crates/permit0-cli/src/cmd/serve.rs:130-134`. It has no equivalent of the local hook's `SqliteSessionStore::get_session` and `record_action` flow in `crates/permit0-cli/src/cmd/hook.rs:547-607`.
**Recommendation:** Either remove remote session-aware behavior from v1 or add daemon session persistence and metadata forwarding to v1.

### Finding 5: CLI help text is already stale for `OpenClaw`

**Severity:** Minor
**Location:** Change 7: Update CLI argument parser
**Claim:** The `--client` help text only needs adding Codex.
**Reality:** `main.rs` help currently lists `claude-code`, `claude-desktop`, and `raw` in `crates/permit0-cli/src/main.rs:59-64`, but `ClientKind::from_str` also supports `openclaw`, `open-claw`, and `open_claw` in `crates/permit0-cli/src/cmd/hook.rs:95-107`.
**Recommendation:** Update the help text to include both existing `openclaw` support and new `codex` support.

## Verified Claims

- `ClientKind` currently has `ClaudeCode`, `ClaudeDesktop`, `OpenClaw`, and `Raw`; adding `Codex` is localized in `crates/permit0-cli/src/cmd/hook.rs:49-68`.
- The current `FromStr` implementation is the right place to add `codex`, `codex-cli`, and `codex_cli` in `crates/permit0-cli/src/cmd/hook.rs:95-107`.
- `HookInput` currently contains only `tool_name` and `tool_input`, so adding optional Codex fields with serde defaults is backward-compatible in `crates/permit0-cli/src/cmd/hook.rs:157-166`.
- `main.rs` parses `PERMIT0_CLIENT` and passes `ClientKind` into `hook::run`, so no new CLI flag is needed for `--client codex` in `crates/permit0-cli/src/main.rs:239-276`.
- The existing remote endpoint builder and response parser can be reused for the HTTP call in `crates/permit0-cli/src/cmd/hook.rs:303-337`.
- The engine API is client-agnostic: `get_permission` accepts a `RawToolCall` and `PermissionCtx` in `crates/permit0-engine/src/engine.rs:103-121`.
- Packs are loaded through the CLI engine factory and do not encode client kind in `crates/permit0-cli/src/engine_factory.rs:69-111`.

## Questions for the Author

1. Should Codex support be implemented as a separate `run_codex` path to keep fail-closed handling obvious?
2. Should remote requests include `metadata` and `client_kind` from day one, even if full session history is deferred?
3. Should the existing remote HITL mismatch be fixed in a preparatory PR before Codex support?
